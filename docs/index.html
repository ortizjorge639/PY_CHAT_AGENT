Chat-Over-Data Bot

Ask plain-English questions about your tabular data — directly inside Microsoft Teams or a built-in web UI.

Last updated: June 9, 2026 · Author: Contoso Data Engineering

Overview

The Chat-Over-Data Bot is a conversational interface that lets users query structured tabular data without writing code or opening a spreadsheet. Ask questions like "How many parts have replacement intent?" or "Show me all rows where confidence is above 0.9" and get instant, accurate answers.

Who it's for

  • Data analysts — query data without SQL
  • Business users — explore insights in Teams
  • Engineers — deploy a no-code pattern in hours, not weeks
  • Stakeholders — understand data through conversation

Key value proposition

  • No code to write — just ask a question
  • Works inside Teams and in a browser
  • Secure by design — LLM never sees raw data
  • Deploy in hours — standard Azure infrastructure
  • Swappable data layer — Excel or SQL Server, no code changes

Tip: The bot routes natural-language questions to typed Python functions. The LLM decides which function to call based on the question; it never directly queries your data. This keeps your data safe and costs low.

Architecture

The bot processes user questions through a multi-layer pipeline: user input → Bot Framework → Microsoft Agent Framework → LLM decision-making → typed data tools → data layer → formatted response.

Request flow

User (Teams or Web UI) │ "How many comments have replacement intent?" ▼ Bot Framework adapter (aiohttp, port 3978) ▼ Microsoft Agent Framework ──► LLM (GPT-4.1 mini) decides which tool to call ▼ Data tools (count_rows, query_table, group_by, list_all, …) ▼ Data layer (Excel or SQL Server) ▼ Formatted answer returned to user "22 out of 25 comments have replacement_intent = 1."

Design principles

  • LLM never sees raw data — The LLM only sees function signatures and must decide which function to call. It never reads table contents directly. This protects your data and reduces token costs.
  • Mini model by design — The bot's job is tool routing, not complex reasoning. GPT-4.1 mini is cheaper, faster, and equally accurate for this workload. You're paying for function selection, not reasoning.
  • Swappable data layer — Switch between Excel and SQL Server with a single environment variable change. No code modifications required.

How tool calling works: The LLM receives a structured description of available functions (their names, parameters, descriptions). It analyzes the user's question and responds with the function name and arguments it wants to execute. The bot calls that function, gets the result, and asks the LLM to compose a human-readable response.

Getting Started — Checkpoint Walkthrough

The bot deployment is structured as four checkpoints. Each is a save point — don't move on until the current one works end-to-end. This keeps the learning curve flat and troubleshooting focused.

Checkpoint 0: Azure Resources

Goal: Verify you have an Azure OpenAI resource with a tool-calling model deployed.

Prerequisites:

  • Azure subscription with permission to create resources
  • Azure OpenAI instance (reuse from a prior project if available)

Key steps:

  • Open Azure Portal → Azure OpenAI resource → Keys and Endpoint
  • Copy the endpoint URL, API key, and deployment name (e.g., gpt-5-mini)
  • Keep these values handy for Checkpoint 1

Status: ✓ Prerequisite

Checkpoint 1: Local Web UI

Goal: Bot answers questions in a browser on your machine.

Prerequisites:

  • Checkpoint 0 complete
  • Python 3.11 installed
  • Azure OpenAI credentials from Checkpoint 0

Key steps:

  • Clone the repo and create a Python virtual environment
  • Copy .env.example to .env and fill in Azure OpenAI credentials
  • Set DATASOURCE=excel (default; requires no extra setup)
  • Run pip install -r requirements.txt then python main.py
  • Open http://localhost:3978 and ask: "How many rows are there?"

See full guide in README →

Checkpoint 2: Deploy to Azure

Goal: Bot runs in Azure App Service; responds through Azure Bot Service.

Prerequisites:

  • Checkpoint 1 complete
  • Azure subscription with sufficient permissions
  • Azure CLI installed

Key steps:

  • Register an app in Entra ID and capture its client ID and secret
  • Create Azure Bot Service and App Service (Linux, Python 3.11)
  • Configure App Service environment variables with Azure OpenAI and Bot credentials
  • Create a zip file and deploy with az webapp deployment source config-zip
  • Point Bot Service messaging endpoint to your App Service URL
  • Test using Bot Service's "Test in Web Chat" feature

See full deployment guide in README →

Checkpoint 3: Teams Integration

Goal: Bot is usable inside Microsoft Teams.

Prerequisites:

  • Checkpoint 2 complete
  • Teams tenant with custom app sideloading enabled

Key steps:

  • Update teams-app/manifest.json with your Entra ID app ID
  • Create a zip containing manifest.json, outline.png, color.png
  • Upload the zip to Teams → Apps → Manage your apps → Upload custom app
  • Add the bot to your chat and ask a question

See Teams setup guide in README →

Checkpoint 4: Secure Access

Goal: Web app is accessible only to approved users; network access is restricted.

Prerequisites:

  • Checkpoint 2 complete
  • Coordination with infrastructure team for network setup

Key steps:

  • Configure network restrictions (App Gateway, access restrictions, or private endpoint)
  • Enable Entra ID authentication (Easy Auth) on App Service
  • Exclude /api/messages from Easy Auth so Bot Service can reach the endpoint
  • Set environment variable REQUIRE_AUTH=true for defense-in-depth
  • Verify authentication works by opening web UI in a private browser window

See security setup guide in README →

Security Model

The bot implements defense-in-depth across four layers:

Four-layer security stack

  1. Network restrictions — App Gateway, access restrictions, or private endpoint limit which IPs can reach the App Service
  2. Entra ID authentication — Easy Auth requires users to sign in with their organizational account
  3. User/group assignment — Only approved users are granted access (enforced at the app level)
  4. Download gating — File exports require membership in a designated security group

Network approach comparison

Choose one based on your infrastructure requirements:

Approach How it works Teams compatible? Effort
Application Gateway Private App Service behind a gateway; gateway accepts Bot Service traffic ✓ Yes High
Access restrictions Public endpoint; allowlist AzureBotService tag + VPN range; deny all else ✓ Yes Low
Private endpoint only No public access; only reachable from within the VNet ✗ No High

Important: Teams integrations require inbound access from Microsoft's cloud infrastructure. A fully private endpoint (no public access) will block Teams and "Test in Web Chat" functionality. Work with your infrastructure team to choose an approach that balances security with Teams connectivity.

Defense-in-depth principle

Tip: The bot implements an app-level REQUIRE_AUTH flag that rejects unauthenticated requests even if Easy Auth is misconfigured. This acts as a safety net — if the authentication layer is accidentally disabled, the bot still enforces access control.

Scaling & Production Readiness

The bot is designed to scale. For higher traffic, consider these changes:

Change What to do Why Effort
Session persistence Add Azure Cache for Redis; update session storage in bot_handler.py In-memory sessions don't survive App Service restarts or scale-outs Medium
Hosting tier Upgrade from B1 (Basic) to P1v2 (Premium) App Service Plan Higher throughput, auto-scaling, deployment slots Low
Container hosting Migrate to Azure Container Apps or AKS; build and push Docker image Finer-grained scaling, multi-region deployment, GitOps High
Data layer optimization Add indexes to SQL tables; cache frequently queried datasets Reduce query latency for large datasets Low–Medium

Important: The bot code doesn't change when you scale. You're only changing infrastructure — App Service plan, hosting model, or caching layer. The core logic, data layer, and Agent Framework integration remain the same.

Troubleshooting

Common issues and their fixes:

Symptom Likely Cause Fix
"Test in Web Chat" — no response, no errors Messaging endpoint wrong or behind App Gateway Verify URL ends with /api/messages; if behind gateway, use gateway URL — not the App Service URL
403 Forbidden on App Service URL App Service behind App Gateway / APIM Use the gateway's public URL instead; direct access is blocked
Unauthorized errors in Bot Service App ID / password / tenant ID mismatch Double-check all three values match between Entra, Bot Service, and App Service env vars
"Test in Web Chat" — HTTP status code Unauthorized Easy Auth blocking /api/messages Add /api/messages to Easy Auth excluded paths using the included fix_easyauth_bot.ps1 script
missing service principal error App Registration has no service principal Azure Portal → Entra ID → Enterprise applications → search your app. If missing, create one via az ad sp create --id <APP_ID>
App Service returns 5xx errors Startup command not set or dependencies missing Check App Service → Log stream; verify startup command is antenv/bin/python main.py
ModuleNotFoundError for local packages Windows backslash paths in zip **or** Oryx extraction issue Re-zip using the Python zipfile script (Step 5 in README); ensure sys.path.insert is in main.py
Could not find requirements.txt in build log Zip structure wrong — requirements.txt nested in subfolder Re-zip from inside the project folder, not its parent
Build finishes in <60 seconds Oryx skipped pip install Verify SCM_DO_BUILD_DURING_DEPLOYMENT=1 is set; use config-zip (not deploy --type zip)
unexpected keyword argument 'tool_choice' Agent Framework version mismatch Pin exact versions in requirements.txt to match bundled wheels; redeploy
Teams upload — Manifest parsing error Placeholder app ID in manifest.json Replace both instances of <your-app-id-from-entra> with the actual Application (client) ID
